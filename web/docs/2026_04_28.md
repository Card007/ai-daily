🥉 **pip 26.1 新特性：锁文件与依赖冷却机制**

[What's new in pip 26.1 - lockfiles and dependency cooldowns!](https://simonwillison.net/2026/Apr/28/pip-261/#atom-everything) — simonwillison.net · 04-28 05:23 · 🛠 工具 / 开源

> pip 26.1 围绕“可复现安装”和“更稳定的依赖解析”做了关键升级，包括引入 lockfiles（锁文件）与 dependency cooldowns（依赖冷却）机制。锁文件让项目能够固定解析结果，减少不同机器/不同时间安装时的漂移，提升部署一致性。依赖冷却则针对频繁变动的依赖解析过程做节制，降低网络抖动、索引更新或解析反复带来的不确定性与成本。该版本同时停止支持已于 10 月进入 EOL 的 Python 3.9，并点名 macOS 仍可能默认带着较旧的 python3 环境，需要开发者留意升级与兼容策略。整体结论是：pip 正在补齐“默认工具也能提供更强工程化确定性”的能力，向更现代的依赖管理体验靠拢。

💡 **为什么值得读**: 想把 Python 依赖安装从“看运气”变成“可复现、可控”，这篇更新解读能帮你快速判断 pip 26.1 是否值得立刻纳入团队工具链。

### 19. 在不同架构上向 C 函数传入过少寄存器参数会带来什么后果

[Looking at consequences of passing too few register parameters to a C function on various architectures](https://devblogs.microsoft.com/oldnewthing/20260427-00/?p=112271) — **devblogs.microsoft.com/oldnewthing** · 04-27 14:00 · ⭐ 20/30

> 核心问题是：调用 C 函数时如果“应该放在寄存器里的参数没有按 ABI/调用约定正确传够”，在不同 CPU 架构上会出现哪些具体后果。寄存器传参架构（如常见的 x86/x64、ARM64 等）里，缺失的参数不会自动补零或报错，callee 仍会从约定的参数寄存器读取值，结果往往是读取到调用点之前遗留的“垃圾值”，导致逻辑错误、数据泄漏或难以复现的崩溃。由于这是 C 语义层面的未定义行为（UB），优化器可能基于“参数一定按约定提供”的假设做激进优化，使得问题在不同编译器/优化级别/链接方式下表现完全不一致。文章特别指出 Itanium（IA-64）因为其更复杂的寄存器与调用机制（寄存器栈/溢出回填等）会让这种“少传寄存器参数”的代价更高、症状更诡异，排查难度也显著上升。结论是：这类错误在任何平台都不可靠且不可移植，而在 Itanium 上尤其“更糟”，应通过严格的函数原型匹配、编译器告警与静态分析来避免。
