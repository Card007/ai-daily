### 10. 开发一个跨进程、读者数量受限的读写锁（第一部分）：信号量

[Developing a cross-process reader/writer lock with limited readers, part 1: A semaphore](https://devblogs.microsoft.com/oldnewthing/20260428-00/?p=112278) — **devblogs.microsoft.com/oldnewthing** · 04-28 14:00 · ⭐ 21/30

> 目标是在多个进程之间实现一个读写锁，并且把并发读者数量限制在固定上限以内。方案把“允许进入读区的名额”抽象成一锅可取用的 token，用 Windows 信号量（semaphore）的计数来表示剩余名额：读者进入时获取 1 个计数、离开时释放 1 个计数，从而天然实现“最多 N 个读者”。写者则通过把信号量的计数一次性耗尽（等价于拿走所有 token）来获得独占访问权，确保写入期间不会再有新读者进入。这个部分重点奠定了用信号量做跨进程资源配额控制的基础，为后续组合出完整读写锁（含公平性/切换细节）做铺垫。结论是：把并发读者上限建模为“可消耗的计数资源”是跨进程场景下实现受限读写并发的一条直接可复用的路径。

🏷️ Windows, semaphore, reader-writer lock, concurrency

### 12. 非法状态 vs 不想要的状态

[Illegal vs Unwanted States](https://buttondown.com/hillelwayne/archive/illegal-vs-unwanted-states/) — **buttondown.com/hillelwayne** · 04-28 15:14 · ⭐ 21/30

> 很多系统状态看起来“绝不该发生”，但实际上更准确的分类是“允许短暂出现、但不希望停留”的不想要状态（unwanted）而非非法状态（illegal）。文章用日历应用的数据建模举例：若事件结构允许同一用户的事件时间段重叠，那么“双重预订”可能被直觉当成非法状态，但在真实业务里它可能只是需要被检测、提示与修复的暂态。把某些状态强行做成非法（例如用更强约束的数据类型彻底禁止）会导致实现复杂度上升，并且把原本应由流程/恢复机制处理的问题挤压到类型系统里，反而降低系统弹性。更合理的做法是区分两类状态：非法状态用类型/约束在构造时杜绝；不想要的状态则设计检测、迁移与纠错路径，让系统能从该状态退出。作者的核心观点是：先判断“能否从该状态恢复、是否需要表达中间过程”，再决定把它建模为 illegal 还是 unwanted，才能在正确性与工程成本之间取得平衡。

🏷️ state machines, invariants, type systems, software design

### 14. 圆弧近似

[Circular arc approximation](https://www.johndcook.com/blog/2026/04/28/circular-arc-approximation/) — **johndcook.com** · 04-28 13:09 · ⭐ 17/30

> 给定圆半径 r 的一段圆弧，只知道整段圆弧的弦长 c，以及“半段圆弧”的弦长 b，目标是估算圆弧长度 rθ（θ 为圆心角）。利用弦长公式 c=2r·sin(θ/2)、b=2r·sin(θ/4)，可以先由 b 求出 sin(θ/4)=b/(2r)，再通过恒等式 sin(θ/2)=2sin(θ/4)cos(θ/4) 把 c 与 b 关联起来，从而反推出 θ。推导中关键一步是把 cos(θ/4) 写成 c/(2b)，进而得到 θ=4·arccos(c/(2b))（或等价的反三角形式），最终圆弧长为 rθ。文章强调：相比只用单个弦长 c 做小角度近似（把弧长近似为弦长），引入半弧弦长 b 能显著减少曲率带来的误差，并给出可直接计算的闭式表达。
