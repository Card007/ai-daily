### 8. Patching and forking in package managers

[Patching and forking in package managers](https://nesbitt.io/2026/05/01/patching-and-forking-in-package-managers.html) — **nesbitt.io** · 05-01 10:00 · ⭐ 21/30

> When an upstream dependency stops responding, maintainers must choose between carrying a patch locally or forking the project to regain control. Patching keeps the relationship with upstream intact and minimizes long-term divergence, but it requires tooling and discipline to track, rebase, and eventually drop patches as upstream changes. Forking can unblock releases quickly and enables deeper changes, but it creates ongoing maintenance burden (security fixes, compatibility updates, and coordination costs) and risks ecosystem fragmentation. The discussion frames the decision as a trade-off between short-term delivery pressure and long-term maintenance liability, with “upstream ghosts you” as the practical trigger. Main takeaway: default to the smallest sustainable intervention (patches) and escalate to a fork only when ownership, responsiveness, or scope makes patch-carrying untenable.

### 9. Developing a cross-process reader/writer lock with limited readers, part 4: Abandonment

[Developing a cross-process reader/writer lock with limited readers, part 4: Abandonment](https://devblogs.microsoft.com/oldnewthing/20260501-00/?p=112291) — **devblogs.microsoft.com/oldnewthing** · 05-01 14:00 · ⭐ 19/30

> Cross-process synchronization has to handle the case where a process dies while it is holding (or partially holding) a reader/writer lock, leaving shared state inconsistent. The design uses an OS-level ownership concept (e.g., a mutex wait that can return an abandonment signal such as WAIT_ABANDONED) so the next waiter can detect that the prior owner exited without releasing correctly. On abandonment, the recovery path treats the lock’s shared bookkeeping (reader counts/slots and writer ownership) as suspect and performs a controlled reset/repair so future acquisitions will not deadlock or permanently lose reader capacity. A key point is that abandoning a lock is not “success”; it is a signal that invariants may be broken, so the algorithm must explicitly re-establish a consistent baseline before letting normal readers/writers proceed. The takeaway is that robust cross-process locks must include an abandonment strategy, not just acquisition/release logic, because process termination is an expected failure mode.
